## Q1 宏内核和微内核是什么？
 内核的概念实际上是操作系统（OS，Operating System）调度软硬件协同工作的方式。
 宏内核（Linux,Android,Unix）和微内核（QNX，Minix,HarmonyOS）就是两套不一样的方式。
 
 **宏内核**：

 所有的核心服务（文件系统、驱动程序、内存管理、网络协议栈）都运行在同一个巨大的“办公室”里——即内核态（Kernel Mode）。
 特点：所有的服务都在一个进程空间，它们之间可以直接通过函数调用来沟通，速度极快。

 **微内核**：

 内核只负责最最基础的任务：进程间通信 (IPC)、基础内存管理和中断处理。其他所有的服务（驱动、文件系统）都被踢出内核，
 作为普通的“用户程序”运行在用户态（User Mode）。
 特点：内核变成了一个高效的“传达室”。如果文件系统要读磁盘，它得先发消息给内核，内核再转发给磁盘驱动程序

 **🦌Deer Says:**
 
  所有操作系统都有两个状态，用户态和内核态，以为硬件提供必要的安全保障。
 
  在宏内核（如 Linux）中，绝大部分操作系统服务（驱动、文件系统、内存管理）都挤在内核态办公。
  
  微内核中，内核被剥离得只剩一个“骨架”（IPC、调度、硬件中断）。文件系统、网卡驱动这些原本在内核里的“高官”，被降级到了用户态。

 *煮个🌰:App 想读文件*：

 App 发起请求，陷入内核态。

 内核把请求转发给“文件系统服务”（跳回用户态）。

 文件系统处理完，再发给内核（又进入内核态）。

 内核转发给“磁盘驱动服务”（再次跳回用户态）。

 代价：这种在用户态和内核态之间反复横跳的行为，会产生大量的**上下文切换（Context Switch）**开销
 
 ## Q2：进程（Process）和线程（Thread）分别是什么？
 首先看一段多进程代码：
```
int main() {
    int x = 100; // 局部变量

    pid_t pid = fork(); // ⚠️ 创建进程：从这里开始，分裂为两个独立世界

    if (pid == 0) {
        // 这是子进程
        x = x + 1;
        printf("子进程: x = %d, 地址 = %p\n", x, &x);
    } else {
        // 这是父进程
        sleep(1); // 等子进程先跑完
        printf("父进程: x = %d, 地址 = %p\n", x, &x);
    }
    return 0;
}
```
运行结果：子进程：x = 101, 地址 = 0x7ffd...；父进程：x = 100, 地址 = 0x7ffd...。虽然它们的虚拟地址一模一样，但子进程修改了 x，父进程的 x 却没变

再看一段多线程代码：
```
int x = 100; // 全局变量，线程间共享

void* my_thread_func(void* arg) {
    x = x + 1; // 修改共享变量
    printf("线程: 我修改了 x 为 %d\n", x);
    return NULL;
}

int main() {
    pthread_t tid;

    printf("主程序: 初始 x = %d\n", x);

    pthread_create(&tid, NULL, my_thread_func, NULL); // ⚠️ 创建线程：大家还在一个屋檐下

    pthread_join(tid, NULL); // 等待线程执行结束

    printf("主程序: 线程结束后的 x = %d\n", x);
    return 0;
}
```
运行结果：主程序：初始 x = 100，线程：我修改了 x 为 101；主程序：线程结束后的 x = 10

所以，进程代码 (fork)	变量 x 物理上变成了两份，线程代码 (pthread)互不干扰	物理上只有一份，大家共用。

***🥑avocado reminds you here:***

但是，需要注意的是***父进程fork()出来的子进程，其中的变量它们的虚拟地址完全一样，但映射到的物理地址（在修改数据后）是不同的***。因为子进程是父进程的完全拷贝。如果父进程的变量 x 存储在 0x400500 这个地址，那么子进程在克隆时，它的代码逻辑、寄存器状态（包括程序计数器 PC 和栈指针 SP）都指向这个地址。所以刚刚fork出子进程（变量的值还没有被改变的时候）。父子进程的页表其实都指向同一个物理地址！
但是，OS 把这两个页表的权限都设为**“只读”**。

当其中一个进程尝试“写”数据时：

由于权限是只读，硬件触发一个 Fault（故障）。

OS 内核介入（ECF 登场！），发现：“哦，你是想改数据啊”。

OS 这才在物理内存里新开辟一块空间，把那 4KB 的数据拷贝过去。

OS 修改出错进程的页表映射，指向这个新地址，并把权限改回“可写”。

重新执行刚才那条写指令。

结果：只有被修改过的那部分内存才会产生物理副本，没改过的（比如代码段）大家一辈子共用


## Q3：子进程和父进程是如何交替运作的（以C手搓shell为例）
老规矩，先上代码：
```
// 解析命令的函数：将 buf 切割存入 argv，从终端输入的命令行被解析存进argv
int parse_line(char *buf, char **argv) {
    char *delim;         // 第一个空格的分隔符
    int argc = 0;        // 参数个数
    int bg;              // 是否在后台运行

    buf[strlen(buf)-1] = ' ';  // 替换换行符为空格
    while (*buf && (*buf == ' ')) buf++; // 跳过开头的空格

    while ((delim = strchr(buf, ' '))) {
        argv[argc++] = buf;
        *delim = '\0';
        buf = delim + 1;
        while (*buf && (*buf == ' ')) buf++;
    }
    argv[argc] = NULL;

    if (argc == 0) return 1;

    // 检查是否以 & 结尾（后台运行）
    if ((bg = (*argv[argc-1] == '&')) != 0)
        argv[--argc] = NULL;

    return bg;
}


void eval(char *cmdline) {
    char *argv[MAXARGS]; 
    char buf[MAXLINE];   
    int bg;              
    pid_t pid;           

    strcpy(buf, cmdline);
    bg = parse_line(buf, argv); 

    if (argv[0] == NULL) return; // 忽略空行

    // 1. 检查是否是内置命令
    if (strcmp(argv[0], "quit") == 0) exit(0);

    // 2. 如果不是内置命令，创建子进程执行
    if ((pid = fork()) == 0) {//如果发现拿的PID是0，说明是子进程
       printf("now I am the son!");// 子进程
        if (execvp(argv[0], argv) < 0) {
            printf("%s: Command not found.\n", argv[0]);
            exit(0);
        }
    }

    // 3. 父进程决定是否等待
    if (!bg) {//bg!=0，如果你的命令末尾有 &，bg 就是 1，否则是 0，所以要是我们输入的代码段后面没有&，一个子进程没执行完父进程就继续等着子进程按照代码的意思干活直到完成为止
        int status;
        if (waitpid(pid, &status, 0) < 0)
            perror("waitpid error");
    } else {
        printf("%d %s", pid, cmdline); // 后台运行，不等待，直接打印 PID
    }
}

int main() {
    char cmdline[MAXLINE];

    while (1) {
        printf("my_shell> "); 
        if (fgets(cmdline, MAXLINE, stdin) == NULL) break;
        eval(cmdline);
    }
}
```

现状：fork() 返回后，父进程和子进程都停在同一行代码。

问题：如果不给它们不同的返回值，它们接下来的行为就会一模一样（比如都去执行 execvp 或者都去 wait），这逻辑就乱了。

解决：内核在恢复这两个进程的执行时，强行往它们的 %rax 寄存器（存储返回值的寄存器） 里塞了不同的值。

给子进程塞个 0。

给父进程塞个 子进程的 PID

可以把 fork() 之后的执行流想象成细胞分裂：

分裂前：一个细胞（进程 A）。

分裂中 (fork)：DNA 完全复制，但分裂完成的一瞬间，内核在两个细胞的“意识”里注入了不同的信息。

分裂后：

子进程醒来发现手里的令牌是 0，于是它知道：“我是替身，我要去换衣服（exec）开始干活了。”

父进程醒来发现手里的令牌是 1234，于是它知道：“我是老板，我得看着那个编号为 1234 的临时工，等它干完活。

:TIPS:这里解释一下这个针对Linux写的shell中的关键作用`execvp`是干啥的：

在 Linux 中，创建一个新程序运行的唯一方式（几乎都是）：先复制一个自己，然后把“自己”抹掉，换成“别人”。

execvp 的具体动作如下：

覆盖记忆：它会把当前进程（子进程）的地址空间全部清空（包括代码、数据、堆栈）。

加载新魂：从磁盘找到你指定的程序（比如 /bin/ls），把它的二进制机器码读入内存。

重置起点：把 CPU 的执行指针（PC）指向新程序的 main 函数开头。

结果：原来的子进程“消失”了，取而代之的是一个正在运行的 ls 程序。但请注意，进程 ID (PID) 没变，只是内核里这个 PID 对应的“灵魂”换了
