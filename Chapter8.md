## Q1 宏内核和微内核是什么？
 内核的概念实际上是操作系统（OS，Operating System）调度软硬件协同工作的方式。
 宏内核（Linux,Android,Unix）和微内核（QNX，Minix,HarmonyOS）就是两套不一样的方式。
 
 **宏内核**：

 所有的核心服务（文件系统、驱动程序、内存管理、网络协议栈）都运行在同一个巨大的“办公室”里——即内核态（Kernel Mode）。
 特点：所有的服务都在一个进程空间，它们之间可以直接通过函数调用来沟通，速度极快。

 **微内核**：

 内核只负责最最基础的任务：进程间通信 (IPC)、基础内存管理和中断处理。其他所有的服务（驱动、文件系统）都被踢出内核，
 作为普通的“用户程序”运行在用户态（User Mode）。
 特点：内核变成了一个高效的“传达室”。如果文件系统要读磁盘，它得先发消息给内核，内核再转发给磁盘驱动程序

 **🦌Deer Says:**
 
  所有操作系统都有两个状态，用户态和内核态，以为硬件提供必要的安全保障。
 
  在宏内核（如 Linux）中，绝大部分操作系统服务（驱动、文件系统、内存管理）都挤在内核态办公。
  
  微内核中，内核被剥离得只剩一个“骨架”（IPC、调度、硬件中断）。文件系统、网卡驱动这些原本在内核里的“高官”，被降级到了用户态。

 *煮个🌰:App 想读文件*：

 App 发起请求，陷入内核态。

 内核把请求转发给“文件系统服务”（跳回用户态）。

 文件系统处理完，再发给内核（又进入内核态）。

 内核转发给“磁盘驱动服务”（再次跳回用户态）。

 代价：这种在用户态和内核态之间反复横跳的行为，会产生大量的**上下文切换（Context Switch）**开销
 
 ## Q2进程（Process）和线程（Thread）分别是什么？
 首先看一段多进程代码：
```
int main() {
    int x = 100; // 局部变量

    pid_t pid = fork(); // ⚠️ 创建进程：从这里开始，分裂为两个独立世界

    if (pid == 0) {
        // 这是子进程
        x = x + 1;
        printf("子进程: x = %d, 地址 = %p\n", x, &x);
    } else {
        // 这是父进程
        sleep(1); // 等子进程先跑完
        printf("父进程: x = %d, 地址 = %p\n", x, &x);
    }
    return 0;
}
```
运行结果：子进程：x = 101, 地址 = 0x7ffd...；父进程：x = 100, 地址 = 0x7ffd...。虽然它们的虚拟地址一模一样，但子进程修改了 x，父进程的 x 却没变

再看一段多线程代码：
```
int x = 100; // 全局变量，线程间共享

void* my_thread_func(void* arg) {
    x = x + 1; // 修改共享变量
    printf("线程: 我修改了 x 为 %d\n", x);
    return NULL;
}

int main() {
    pthread_t tid;

    printf("主程序: 初始 x = %d\n", x);

    pthread_create(&tid, NULL, my_thread_func, NULL); // ⚠️ 创建线程：大家还在一个屋檐下

    pthread_join(tid, NULL); // 等待线程执行结束

    printf("主程序: 线程结束后的 x = %d\n", x);
    return 0;
}
```
运行结果：主程序：初始 x = 100，线程：我修改了 x 为 101；主程序：线程结束后的 x = 10

所以，进程代码 (fork)	变量 x 物理上变成了两份，线程代码 (pthread)互不干扰	物理上只有一份，大家共用。

***🥑avocado reminds you here:***

但是，需要注意的是***父进程fork()出来的子进程，其中的变量它们的虚拟地址完全一样，但映射到的物理地址（在修改数据后）是不同的***。因为子进程是父进程的完全拷贝。如果父进程的变量 x 存储在 0x400500 这个地址，那么子进程在克隆时，它的代码逻辑、寄存器状态（包括程序计数器 PC 和栈指针 SP）都指向这个地址。所以刚刚fork出子进程（变量的值还没有被改变的时候）。父子进程的页表其实都指向同一个物理地址！
但是，OS 把这两个页表的权限都设为**“只读”**。

当其中一个进程尝试“写”数据时：

由于权限是只读，硬件触发一个 Fault（故障）。

OS 内核介入（ECF 登场！），发现：“哦，你是想改数据啊”。

OS 这才在物理内存里新开辟一块空间，把那 4KB 的数据拷贝过去。

OS 修改出错进程的页表映射，指向这个新地址，并把权限改回“可写”。

重新执行刚才那条写指令。

结果：只有被修改过的那部分内存才会产生物理副本，没改过的（比如代码段）大家一辈子共用
